# Ralplan 独立版


- **RALPLAN-DR 结构化审议**
- **Planner → Architect → Critic** 共识工作流
- **严格的 Architect 后 Critic 串行顺序**
- **当 Critic 未批准时最多 5 轮迭代**
- **可选的交互式检查点**
- **先将模糊执行请求导入规划的执行前闸门**
- **只有计划获批后才允许执行交接**

这个技能用于**规划**，不是实现。

---

## 用法

```text
/ralplan "任务描述"
```

### 参数选择规则

- 如果用户显式传入 `--interactive`，则直接启用交互式模式。
- 如果用户显式传入 `--deliberate`，则直接启用审慎模式。
- 如果用户**没有**传入这两个参数，则必须先使用 `AskUserQuestion` 询问用户是否启用它们，并简要解释两者作用后再继续。
- 如果用户只传入了其中一个参数，则只对另一个未提供的参数发起询问。

建议向用户说明：
- `interactive`：在关键检查点暂停，让用户审阅草案并决定是否继续、修改或批准最终计划。
- `deliberate`：适用于高风险任务，会增加失败预演与更完整的测试规划，使计划更谨慎、更重验证。

---

## 原生 Claude Code agent 角色

这个技能设计了三个专用角色：
- **Planner**：负责产出初始计划和 RALPLAN-DR 摘要
- **Architect**：负责对计划进行架构合理性审查
- **Critic**：负责对计划进行质量与验证标准评估

### 使用的本地代理

目标环境必须存在以下本地代理：

| Ralplan 角色 | 使用的本地代理 |
|---|---|
| Planner | `planner-ralplan` |
| Architect | `architect-ralplan` |
| Critic | `critic-ralplan` |

---

## 行为

共识工作流如下：

1. **Planner** 先产出初始计划和紧凑的 **RALPLAN-DR 摘要**。
2. **用户反馈**（仅交互模式）可要求修改或跳过评审。
3. **Architect** 对计划进行架构合理性审查。
4. **Critic** 对审查后的计划进行质量与验证标准评估。
5. **再评审循环** 会持续进行，直到 Critic 返回 `APPROVE`，或达到 5 轮上限。
6. **最终批准**（仅交互模式）向用户呈现执行选项。

## 独立状态协议

该技能必须把自己的运行状态持久化到**唯一的本地文件**：

- 状态文件：`.claude/ralplan/state.json`
- 作用域：仅当前项目
- 依赖：只允许使用 Claude Code 原生能力与 Python 标准库

### 状态字段

状态文件必须始终是一个 JSON 对象，并且至少包含以下字段：

- `active`：布尔值，表示当前 ralplan 工作流是否仍在运行
- `stage`：字符串，表示当前所处阶段
- `interactive`：布尔值，表示是否启用交互式检查点
- `deliberate`：布尔值，表示是否启用审慎模式
- `iteration`：整数，表示当前共识循环轮次，从 0 开始
- `task_summary`：字符串，概述当前规划任务
- `session_id`：字符串，记录当前 Claude 会话标识；如果环境未提供，则保留空字符串，但字段不能缺失
- `updated_at`：字符串，UTC ISO-8601 时间戳，表示最后一次更新状态的时间

### 必须支持的阶段

`stage` 必须且只能使用以下显式阶段名来表达主流程：

1. `init`
2. `awaiting-flags`
3. `planner`
4. `interactive-draft-gate`
5. `architect`
6. `critic`
7. `revision-loop`
8. `interactive-final-gate`
9. `final-output`
10. `completed`

### 阶段迁移要求

技能在执行时必须显式初始化、更新并完成状态，最少满足以下迁移规则：

- 技能刚开始时，初始化状态文件并写入 `stage: "init"`
- 当需要通过 `AskUserQuestion` 询问 `--interactive` / `--deliberate` 时，切换到 `awaiting-flags`
- 进入 Planner 生成草案前，切换到 `planner`
- 如果启用了交互式草案检查点，在向用户展示草案并等待选择时，切换到 `interactive-draft-gate`
- 进入 Architect 串行审查前，切换到 `architect`
- Architect 完成后、启动 Critic 前，再切换到 `critic`
- 当 Critic 未批准且回到 Planner 修订时，切换到 `revision-loop`，并相应增加 `iteration`
- 如果启用了交互式最终批准检查点，在等待用户最终执行选择时，切换到 `interactive-final-gate`
- 当技能输出最终已批准计划或执行简报时，切换到 `final-output`
- 当技能正常结束、用户拒绝、用户选择跳过后直接结束，或停止在最终交付后，必须写入 `completed` 并把 `active` 设为 `false`

### 状态写入规则

- 每次阶段变化都必须立刻更新 `.claude/ralplan/state.json`
- `updated_at` 每次写入时都必须刷新
- `task_summary` 在任务明确后应保持稳定，仅在任务本身被用户改写时更新
- `interactive` 和 `deliberate` 一旦确定，就应在后续状态中持续保留
- `iteration` 在首次 Planner 草案前可以是 `0`；每进入一次“Critic 未批准后的完整修订循环”就加一
- 如果技能因依赖角色代理缺失而无法继续，也必须先把状态写到能解释当前停点的最后阶段，再以 `completed` 收尾并将 `active` 设为 `false`

### 状态与 hook 协作规则

- Stop hook 只负责阻止未完成阶段的提前结束
- SessionStart hook 只负责在会话恢复时提示未完成状态
- UserPromptSubmit hook 只负责初始化状态
- 真正的阶段推进必须由这个技能自己完成

### 推荐初始化 / 收尾方式

如果环境启用了本插件附带的本地 hooks，则应由这些 hooks 协助完成最基本的状态初始化与收尾；但即使 hooks 未启用，技能本身也必须按上述协议自行维护状态。

## Planner 必须输出的内容

Planner 产出的计划必须包含：

### RALPLAN-DR 摘要

- **原则**：3–5 条指导原则
- **决策驱动因素**：前 3 个决策驱动因素
- **可行方案**：至少 2 个可行方案，并附上边界清晰的优缺点
- 如果最后只剩 1 个可行方案，必须包含对被排除方案的**明确失效理由**

### 完整计划正文

计划还必须包含：
- 问题界定
- 假设
- 范围 / 非目标
- 拟议方案
- 实施阶段或工作流
- 风险与缓解措施
- 验证策略
- 验收标准

### 审慎模式附加项

如果启用审慎模式，Planner 还必须额外包含：
- **失败预演**：3 个具体失败场景
- **扩展测试计划**，覆盖：
  - 单元测试
  - 集成测试
  - 端到端测试
  - 可观测性 / 诊断

---

## Architect 必须输出的内容

Architect 做架构合理性审查时必须提供：

- 针对当前计划的**最强反方论证**
- 至少 **一个真实权衡张力**
- 在可行时提供 **综合方案**
- 若为审慎模式，明确指出原则违背标记
- 一个结论：`APPROVE`、`ITERATE` 或 `REJECT`

Architect 不能只是复述计划。
Architect 必须主动挑战计划。

---

## Critic 必须输出的内容

Critic 在评估计划时必须强制检查以下内容：

- 原则与方案是否一致
- 对备选方案的公平评估
- 风险缓解是否清晰
- 验收标准是否可测试
- 验证步骤是否具体

### 审慎模式硬性要求

在审慎模式下，如果以下内容缺失或薄弱，Critic 必须拒绝：
- 失败预演质量
- 扩展测试计划质量

### Critic 结论

Critic 最终必须且只能返回以下之一：
- `APPROVE`
- `ITERATE`
- `REJECT`

Critic 还应提供：
- 结论
- 阻塞问题
- 建议修订
- 验证缺口

---

## 严格顺序规则

> **Architect 和 Critic 必须串行执行，绝不能并行。**

这是强制要求。

**不要**把两者放进同一个并行批次。
必须先等待 Architect 完成，再启动 Critic。

---

## 再评审循环

如果 Critic 返回任何非 `APPROVE` 的结论，必须执行如下闭环：

1. 收集 Architect 和 Critic 的反馈
2. 让 Planner 修订计划
3. 再次运行 Architect 审查
4. 再次运行 Critic 评估
5. 持续重复，直到：
   - Critic 返回 `APPROVE`，或
   - 总轮数达到 5 次

如果 5 轮后仍未批准：
- 向用户展示**当前最佳版本**
- 明确说明未解决的问题
- 不要假装已经达成共识

---

### 交互模式检查点

### 检查点 1：草案评审

如果启用了 `--interactive`，在初始 Planner 输出之后，使用 `AskUserQuestion` 展示：
- 草案摘要
- 原则
- 决策驱动因素
- 可行方案

必须且只能提供以下选项：
1. **继续进入评审**
2. **请求修改**
3. **跳过评审**

行为如下：
- **继续进入评审** → 继续进入 Architect 和 Critic
- **请求修改** → 收集用户修改意见，交给 Planner 修订后再继续
- **跳过评审** → 直接输出草案并结束

### 检查点 2：最终批准

如果启用了 `--interactive` 且 Critic 已批准，则再次使用 `AskUserQuestion`。

展示给用户的最终计划必须包含 ADR 部分：
- 决策
- 驱动因素
- 备选方案
- 选择原因
- 影响
- 后续事项

必须且只能提供以下选项：
1. **批准并通过协作团队实现（推荐）**
2. **批准并直接执行**
3. **清空上下文后实现**
4. **请求修改**
5. **拒绝**

行为如下：
- **批准并通过协作团队实现（推荐）** → 当用户希望并行执行时，使用 `TeamCreate`、`TaskCreate` 和 `Agent` 代理成员创建原生 Claude Code 协作团队工作流
- **批准并直接执行** → 退出规划，并告知用户该计划已批准，可在当前会话中直接实现
- **清空上下文后实现** → 对批准后的计划做干净摘要，为一次新的实现会话准备执行简报
- **请求修改** → 回到 Planner 修订循环
- **拒绝** → 停止

## 原生 Claude Code 执行交接

原始 ralplan 的设计是：规划完成后，可以进入并行执行路径或单代理执行路径，而不是把“规划”伪装成“静默实现”。

这个原生 Claude Code 独立版保留了这一意图，并映射为：

| 规划结果 | 原生 Claude Code 对应方式 |
|---|---|
| 并行执行路径 | `TeamCreate` + `TaskCreate` + `Agent` 代理成员 |
| 单代理执行路径 | 计划批准后继续在当前会话中执行 |
| 清空上下文后实现 | 输出一份适合新会话使用的干净执行简报 |

### 重要规则

这个技能仍然是一个**规划技能**。
除非用户明确选择了某种执行选项，或明确要求在计划交付后继续实现，否则它**不能悄悄切换到实现阶段**。

## 执行前闸门

### 为什么需要这个闸门

如果执行请求过于模糊，就会浪费工作量。
这个规划闸门会拦截那些规格不足的执行请求，并先把它们导入共识规划流程。

这样做可以确保：
- **范围明确**：工作边界清晰可控
- **测试先行定义**：在编码前就定义好成功标准
- **先达成共识**：执行前先让计划接受挑战
- **避免无效实现**：实现从明确目标开始，而不是靠猜

### 好提示词 vs 坏提示词

**可以直接通过闸门**（已经足够具体，可直接执行）：
- `修复 src/hooks/bridge.ts:326 的空值判断`
- `实现 issue #42`
- `为 processKeywordDetector 函数添加校验`
- `执行：\n1. 添加输入校验\n2. 编写测试\n3. 更新 README`
- `在 src/models/user.ts 中添加 user model`

**会被拦截并先进入规划**：
- `修一下这个`
- `把应用做出来`
- `优化性能`
- `添加认证`
- `改得更好一点`

**绕过闸门**：
- `force: 重构 auth 模块`
- `! 全面优化`

### 什么情况下不会触发闸门

只要检测到**任意一个**具体锚点，闸门就会自动放行：

| 信号类型 | 示例提示词 | 为什么会放行 |
|---|---|---|
| 文件路径 | `修复 src/hooks/bridge.ts` | 目标文件明确 |
| Issue/PR 编号 | `实现 #42` | 工作项明确 |
| 驼峰命名符号 | `修复 processKeywordDetector` | 函数明确 |
| 大驼峰命名符号 | `更新 UserModel` | 类明确 |
| 下划线命名符号 | `修复 user_model` | 标识符明确 |
| 测试命令 | `npm test && 修复失败项` | 验证目标明确 |
| 编号步骤 | `执行：\n1. 添加 X\n2. 测试 Y` | 交付物结构明确 |
| 验收标准 | `添加登录 - 验收标准：...` | 成功定义明确 |
| 错误引用 | `修复 auth 中的 TypeError` | 故障对象明确 |
| 代码块 | `添加：\`\`\`ts ... \`\`\`` | 实现锚点明确 |
| 转义前缀 | `force: 直接做` / `! 直接做` | 用户明确要求绕过 |

### 闸门启发式规则

当满足以下条件时，应将请求视为规格不足：
- 它带有执行意图
- 它没有上表中的任何具体锚点
- 并且内容足够短或足够模糊，以至于执行仍然需要大量范围探索

如果出现这种情况，应先把请求导入当前规划工作流。

## 端到端流程示例

1. 用户说：`add user authentication`
2. 技能初始化状态文件，并写入 `stage=init`
3. 如果尚未确定 `interactive` / `deliberate`，则切换到 `stage=awaiting-flags` 并先提问
4. 闸门检测到：这是一个带有执行意图但缺少文件、符号、issue、测试目标和验收标准的模糊请求
5. 技能将其重定向到 RALPLAN-DR 规划流程，并在进入 Planner 前写入 `stage=planner`
6. Planner 产出初始计划
7. 若启用交互模式，则在草案评审前写入 `stage=interactive-draft-gate`
8. 进入 Architect 前写入 `stage=architect`
9. 进入 Critic 前写入 `stage=critic`
10. 若 Critic 未批准，则写入 `stage=revision-loop` 并递增 `iteration`
11. 若计划获批且启用交互模式，则在最终执行选项前写入 `stage=interactive-final-gate`
12. 在输出最终计划或执行交接前写入 `stage=final-output`
13. 完成后写入 `stage=completed` 且 `active=false`

## 故障排查

| 问题 | 解决方式 |
|-------|----------|
| 明明提示词已经很具体，却仍然触发闸门 | 增加文件引用、函数名、issue 编号或验收标准 |
| 想绕过闸门 | 使用 `force:` 或 `!` 前缀 |
| 模糊提示词没有触发闸门 | 手动应用“规格不足”启发式，并主动进入规划 |
| 用户想跳过规划 | 如果用户明确坚持，可以承认绕过，但要说明这会失去共识校验 |
| 会话中断后恢复 | 读取 `.claude/ralplan/state.json`，按 SessionStart hook 注入的剩余阶段提示继续 |

## 本技能的执行说明

当这个技能被调用时，严格按以下流程执行。

### 第 1 步：解析输入并初始化状态

提取：
- 任务描述
- 是否包含 `--interactive`
- 是否包含 `--deliberate`

并且必须立即初始化 `.claude/ralplan/state.json`，至少写入：
- `active: true`
- `stage: "init"`
- `interactive`
- `deliberate`
- `iteration: 0`
- `task_summary`
- `session_id`
- `updated_at`

### 第 2 步：必要时询问参数选择

如果用户没有显式传入 `--interactive` 或 `--deliberate`，必须先使用 `AskUserQuestion` 询问，并在发起询问前把状态切换到 `awaiting-flags`。

- 如果两个参数都缺失，则一次性询问这两个参数。
- 如果只缺失其中一个参数，则只询问缺失的那个参数。

询问时必须解释：
- `interactive`：是否在草案评审与最终批准这两个检查点暂停，交由用户决定后续动作。
- `deliberate`：是否启用更审慎的规划模式，增加失败预演与扩展测试计划，适合高风险任务。

如果用户已显式传入其中某个参数，则对该参数不再提问，直接按用户输入处理。

### 第 3 步：判断是否启用审慎模式

满足以下任一条件时启用审慎模式：
- 显式带有该参数
- 用户在 `AskUserQuestion` 中选择启用该模式

一旦参数选择确定，后续所有状态写入都必须保留最终确定的 `interactive` 与 `deliberate` 值。

### 第 4 步：应用执行前闸门

如果请求过于模糊，不适合安全执行，应明确说明你正在先将它导入共识规划流程。

### 第 5 步：产出初始计划

进入 Planner 之前，必须把状态切换到 `planner`。

使用一个 `Agent` 调用来承担 Planner 角色。

必须使用本地 `planner-ralplan` 代理。

Planner 的代理契约必须要求其输出所有强制部分。

### 第 6 步：交互式检查点 1

如果启用了交互式模式，向用户展示草案前必须把状态切换到 `interactive-draft-gate`，然后询问用户是否要：
- 继续进入评审
- 请求修改
- 跳过评审

并严格按上文定义的分支处理。

如果用户请求修改并回到 Planner，则先切换到 `revision-loop`，必要时更新 `task_summary`，然后再回到 `planner`。

如果用户选择跳过评审并直接输出草案，则先切换到 `final-output`，完成输出后必须再写入 `completed` 并将 `active` 设为 `false`。

### 第 7 步：运行 Architect 审查

进入 Architect 前，必须把状态切换到 `architect`。

使用一个 `Agent` 调用来承担 Architect 角色。

必须使用本地 `architect-ralplan` 代理。

此时**不要**启动 Critic。
必须先等待 Architect 完成。

### 第 8 步：运行 Critic 审查

Architect 完成后、启动 Critic 之前，必须把状态切换到 `critic`。

使用一个 `Agent` 调用来承担 Critic 角色。

必须使用本地 `critic-ralplan` 代理。

Critic 必须返回以下之一：
- `APPROVE`
- `ITERATE`
- `REJECT`

### 第 9 步：必要时进入迭代

对于任何非 `APPROVE` 结论，都要先把状态切换到 `revision-loop`，再让 Planner 修订并重新进入完整循环。
最多允许 5 轮迭代。

每进入一次“Critic 未批准后的完整修订循环”，都必须增加 `iteration`。

### 第 10 步：交付批准后的计划

如果不是交互式模式：
- 先把状态切换到 `final-output`
- 输出最终计划
- 包含 ADR
- 然后写入 `completed` 并停止

如果是交互式模式：
- 在向用户提出最终批准问题之前，必须把状态切换到 `interactive-final-gate`
- 按用户选择的执行交接路径继续
- 一旦进入最终交付消息，切换到 `final-output`
- 在技能停止前，必须写入 `completed` 并将 `active` 设为 `false`

如果专用角色代理不可用、用户明确拒绝、或技能在最终交付后结束，也必须以 `completed` 作为最后状态。

## Planner 提示词模板

当你调用 `planner-ralplan` 时，应确保它遵循以下输出契约：

```text
你是一个共识规划工作流中的 Planner。

请为用户任务创建一个实施计划。

你必须输出：
1. RALPLAN-DR 摘要
   - 原则（3-5）
   - 决策驱动因素（前 3 个）
   - 可行方案（至少 2 个）以及边界清晰的优缺点
   - 如果最终只剩 1 个可行方案，说明为什么其他方案无效
   - 如果启用了审慎模式：补充失败预演（3 个场景）和扩展测试计划（单元测试 / 集成测试 / 端到端测试 / 可观测性）

2. 完整计划
   - 问题界定
   - 假设
   - 范围 / 非目标
   - 拟议方案
   - 实施阶段或工作流
   - 风险与缓解措施
   - 验收标准
   - 验证计划

要求内容具体、平衡，并面向实施。
```

```text

- APPROVE
- ITERATE
- REJECT

- 阻塞问题
- 建议修订
- 验证缺口
```

## 最终输出要求

最终计划必须包含一个 **ADR 部分**：

- **决策**
- **驱动因素**
- **备选方案**
- **选择原因**
- **影响**
- **后续事项**

不要省略 ADR。

## 不可妥协的要求

- 保留 Planner → Architect → Critic 的语义
- 保留 Architect 之后再 Critic 的串行执行顺序
- 保留最多 5 轮的共识循环
- 保留交互式检查点行为
- 保留审慎模式规则
- 保留执行前闸门行为
- 保留执行交接意图
- 不依赖任何非原生 Claude Code 的技能、运行状态或挂钩

如果某个专用角色代理不可用，则不应继续执行该技能。
